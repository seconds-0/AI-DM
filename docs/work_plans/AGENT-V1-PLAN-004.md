## Problem Statement

Implement the logic within the Planner flow to write the analysis results (generated by the Gemini model in PLAN-002) to the appropriate `PlannerState` document in Firestore using native Genkit Firestore utilities (`update`). Additionally, update the status of the processed `PlannerTask` document.

## Components Involved

*   `src/flows/planner.flow.ts` (Update flow logic)
*   `@genkit-ai/firebase` plugin (Firestore utilities)
*   `update` function from Genkit
*   `src/data/schemas.ts` (Interfaces: `PlannerState`, `PlannerTask`)
*   Firestore Database

## Proposed Solution / Design Approach

1.  **Locate Target Documents:** Identify the paths for the `PlannerState` document (linked from the `Campaign` document, fetched in PLAN-003) and the specific `PlannerTask` document that triggered the flow (path derived from the event data in PLAN-001).
2.  **Prepare `PlannerState` Update:** Based on the structured suggestions received from Gemini (PLAN-002), prepare the necessary updates to the `PlannerState` document. This might involve merging new plot vectors, updating NPC intents, adding analysis logs, etc. Update the `lastUpdatedAt` timestamp.
3.  **Update `PlannerState`:** Use `await update({ path: plannerStatePath, value: updatedPlannerState, schema: PlannerState })` to write the changes back to Firestore. Consider using `updateMask` or merging strategies if only partial updates are desired, but a full overwrite might be simpler initially.
4.  **Update `PlannerTask` Status:** Update the status field of the triggering `PlannerTask` document to `completed` (or `failed` if an error occurred during processing). Use `await update({ path: taskPath, value: { status: 'completed' }, schema: PlannerTask /* Specify schema? */ })`, potentially using `merge: true` if supported, or fetching the task, updating status, and writing back.
5.  **Error Handling & Logging:** Wrap Firestore `update` calls in try/catch blocks and log success/failure.

## Questions / Uncertainties

*   Best strategy for updating `PlannerState` (overwrite vs. merge)? Overwrite is simpler if the LLM generates the complete desired state based on context.
*   Does Genkit `update` support `merge: true` or require full document replacement? (If replacement, need to ensure `plannerState` object is up-to-date before merging suggestions).
*   Handling potential race conditions if multiple Planner instances are triggered for the same campaign? (Unlikely with Eventarc triggers for specific task documents, but PlannerState updates could potentially conflict. Assume last-write-wins for V1).

## Acceptable Tradeoffs

*   Overwriting `PlannerState` document instead of merging fields initially.
*   Basic error handling for write failures.
*   Assuming no significant race conditions on `PlannerState` for V1.
